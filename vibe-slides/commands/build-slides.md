# /vibe-slides:build-slides

Compile the project's `slides.md` into `out.pptx` by writing a pptxgenjs script.

This command is agent-driven: you (the agent) read the inputs, compose a
`build.js`, and run it. There is no deterministic compiler — the point of
this skill is that you get some authorial latitude within the project's
rules.

---

## Before you start

The working directory is the project root. Confirm these files exist:

- `slides.md` — the deck content
- `theme.json` — the palette/typography
- `components/_manifest.json` + one `.js` file per component listed there
- `vibe-slides.md` — **the project's style rules. READ THIS FIRST, every build.**
- [`experiences.md`](../experiences.md) (this plugin's root) — **the user's accumulated agent-tuning lessons across projects. READ EVERY BUILD, no exceptions.** Binding unless project `vibe-slides.md` overrides.

If any of those is missing, stop and report which — don't guess defaults.

### Resolving plugin paths

The build.js example below references this plugin's `lib/render_helpers.js`.
Resolve `<PLUGIN_ROOT>` as the root of this installed plugin. In Claude Code,
this command usually finds it:

```bash
ls -d ~/.claude/plugins/cache/*/vibe-slides/*/ 2>/dev/null | sort -V | tail -1
```

In Codex, resolve it from the installed skill/plugin path. Substitute the
absolute path for `<PLUGIN_ROOT>` everywhere below before writing build.js.

## Workflow

### Step 1 — Parse `slides.md` into a slide list

You do this in your head while reading. For each slide, capture:

- **layout** (required)
- **title, subtitle, and any layout-specific front-matter keys**
- **body content** — bullets, prose, column splits (`::column`), images
- **inline component calls** — each `::name key=value` line, plus body if
  it has `::end`

Do NOT emit an intermediate JSON or AST file. The agent holds the parse
mentally and writes build.js directly.

### Step 2 — Plan the deck

Before writing code, sketch mentally:

- Total slide count.
- For each slide, which layout helper or custom arrangement it needs.
- Which components get called on each slide.
- Whether you're introducing any repeating motif (e.g. numbered circles
  on every step-slide) — these should be applied consistently once chosen.

If any slide's layout isn't clear (e.g., `canvas` slide with vague
intent), ask the user a single clarifying question rather than guessing.

### Step 3 — Write `build.js`

Create `<project>/build.js`. Structure it like this:

```javascript
// build.js — generated by /vibe-slides:build-slides
const path = require("path");
const {
  createDeck,
  addSlideTitle,
  addFooter,
} = require("<PLUGIN_ROOT>/lib/render_helpers.js");  // substitute the absolute path discovered above

const PROJECT_DIR = __dirname;
const { pres, theme, components, dims } = createDeck(PROJECT_DIR);

// Convenience aliases
const { callout, arrow, stat, num_circle /* ... */ } = components;
const W = dims.w, H = dims.h;

// --- Slide 1 ------------------------------------------------------------
function slide1_title() {
  const s = pres.addSlide();
  s.background = { color: theme.color.primaryDeep };
  // ...
}

// --- Slide 2 ------------------------------------------------------------
function slide2_motivation() {
  const s = pres.addSlide();
  s.background = { color: theme.color.bg };
  addSlideTitle(s, theme, "Why Formal Methods Don't Scale",
                "And why LLMs change the equation");
  callout(s, { x: 0.7, y: 1.9, w: 5.8, h: 4.8,
               title: "The Bottleneck",
               body: "Classical verifiers need hand-written specs...",
               variant: "warn" });
  callout(s, { x: 6.85, y: 1.9, w: 5.8, h: 4.8,
               title: "Our Bet",
               body: "Let the LLM write the specs...",
               variant: "info" });
  addFooter(s, theme, 2, 12);
}

// ...

// Build the deck
slide1_title();
slide2_motivation();
// ...

pres.writeFile({ fileName: path.join(PROJECT_DIR, "out.pptx") })
    .then(f => console.log("wrote:", f))
    .catch(e => { console.error(e); process.exit(1); });
```

**Naming convention:** one function per slide, `slide<N>_<short-handle>`.
Call them in order at the bottom. This makes the build script itself
read like a table of contents.

### Step 3b — Image stubs

If a slide uses `image: STUB` in front matter or has an `::image-stub`
inline component, it's a placeholder for an image that doesn't exist
yet.

1. **Render the stub** at the intended position:

   ```javascript
   components.image_stub(s, { x, y, w, h,
                              caption: "Pipeline",
                              description: fullDescription });
   ```

2. **Append the full description to the slide's speaker notes** — the
   box only shows a truncated preview. Collect all stub descriptions
   for a slide, then call `s.addNotes(...)` once at the end of that
   slide's function:

   ```javascript
   const notes = [];
   if (frontMatterNotes) notes.push(frontMatterNotes);
   const desc = "A horizontal flow diagram showing the compilation pipeline...";
   components.image_stub(s, { x: 7, y: 2, w: 5.5, h: 4.5,
                              caption: "Pipeline", description: desc });
   notes.push(`[IMAGE STUB — Pipeline]\n${desc}`);
   // ...
   if (notes.length) s.addNotes(notes.join("\n\n---\n\n"));
   ```

3. If multiple stubs appear on the same slide, push one entry per stub
   and separate with a `---` divider so they stay readable in presenter
   view.

4. Stubs are deliberately temporary. Never auto-generate a real image —
   the point is to reserve the spot and keep the description alive
   until the user produces the asset.

### Step 4 — Hard rules while writing build.js

These are non-negotiable. Violating any is a bug.

1. **Never hardcode a hex color.** Always `theme.color.xxx`.
2. **Never hardcode a font name.** Always `theme.font.xxx`.
3. **Never set `fontSize` to a literal number.** Always `theme.size.xxx`.
   If the project needs a size theme doesn't define, ask the user to add it.
4. **Never re-implement a component inline.** If you need a callout, call
   `components.callout()`. If the library is missing a primitive you'd
   need more than once, stop and ask whether to add it to the project's
   components/ directory rather than inlining.
5. **Respect `vibe-slides.md`.** If project rules disallow something (e.g. "no
   drop shadows"), override the component default by passing a plainer
   props set if the component allows it, or use a different component.
6. **Honor margins.** Keep every content element ≥ `theme.space.margin`
   from the slide edges (decorative bleeds are the only exception).
7. **One title per content slide.** Use `addSlideTitle`.
8. **Pages count from 1.** Page 1 is the title slide and typically has
   no footer.

### Step 5 — Run the build

```bash
# pptxgenjs is installed globally; set NODE_PATH so require() finds it.
NODE_PATH=$(npm root -g) node <project>/build.js
```

If it errors, read the error, fix, re-run. Do not create a new file —
iterate on build.js.

### Step 6 — Visual QA (MANDATORY)

First render is rarely correct. Before declaring done:

1. Convert to images:

   ```bash
   # requires LibreOffice + Poppler
   python <PPTX_SKILL_ROOT>/scripts/office/soffice.py \
       --headless --convert-to pdf <project>/out.pptx
   pdftoppm -jpeg -r 150 <project>/out.pdf <project>/slide
   ```

   If `soffice` isn't available on this system, fall back to **content QA
   only** (python-pptx text extraction + geometry bounds check) and
   clearly tell the user visual QA was skipped.

2. Spawn a subagent with fresh eyes (use the Agent tool, Explore or
   general-purpose subagent) and give it the prompt in
   an installed `pptx` skill's `SKILL.md` under "Visual QA". Feed it the
   rendered JPGs with a one-line expected description per slide.

3. For every issue the subagent reports, fix in build.js, re-run
   node, re-convert, re-inspect — until a clean pass.

4. **Never declare success without at least one fix-and-verify cycle.**

### Step 7 — Report

When done, summarize for the user:

- Path to `out.pptx`.
- Any vibe-slides.md constraints that came close to being violated and how
  you resolved them.
- Any slides where you made an authorial judgment call (e.g. chose a
  layout for a vague canvas slide) — briefly, so they can push back.
- Whether visual QA passed fully, partially, or was skipped (with
  reason).

## Authorial latitude — what you CAN vary

You have creative freedom in these places:

- **Canvas slide layouts.** Markdown describes intent; you decide
  placement. Prefer symmetry, clear alignment, and the motifs declared
  in vibe-slides.md.
- **Variant choice on components.** If a callout's variant isn't
  specified, pick one that fits the slide's tone (warn for problems,
  ok for solutions, info for neutral notes).
- **Adding motif elements.** If vibe-slides.md declares a motif (e.g.
  numbered circles), apply it consistently even if not every bullet
  explicitly calls for it — that's the point of a motif.
- **Sizing within constraints.** If `w`/`h` isn't given to a component,
  you choose; just keep it proportional to the slide and consistent
  across similar slides.

## What you CANNOT do

- Invent new components on the fly. If the library lacks what you
  need, stop and ask.
- Deviate from the markdown's information order. You can rearrange
  visual layout but not reshuffle content.
- Ignore vibe-slides.md rules because they're inconvenient.
- Skip visual QA when tools are available.

## Quick reference — component call pattern

```javascript
// Every component takes (slide, propsObject).
components.callout(s, { x: 0.7, y: 2.0, w: 6, h: 1.2,
                        title: "{P}", body: "caller's pre-condition",
                        variant: "info" });

components.arrow(s, { from: [2.7, 2.85], to: [2.7, 3.3] });

components.stat(s, { x: 1, y: 2, w: 2.9, h: 2.0,
                     number: "143K", label: "lines of C" });

components.num_circle(s, { x: 0.7, y: 2.0, d: 0.7, n: 1 });
```

Look up exact param names in `components/_manifest.json` before calling;
don't assume.
